# APK Threat Analysis Platform

> **CIDecode Hackathon 2026** — Problem Statement 02  
> CID Karnataka & CCITR, Bengaluru | 4–5 July 2026

A web-based APK threat analysis platform purpose-built for cybercrime investigation. Upload a suspicious Android APK and receive a complete threat report — what it does, what data it targets, which servers it contacts, and a visual map of the criminal infrastructure behind it.

---

## The Problem

Malicious Android applications are a rapidly growing vector for fraud and cybercrime in India. These APKs actively conceal their behavior — using anti-analysis protections, obfuscation, and delayed execution — making them difficult to detect with static tools alone. Law enforcement investigators need deeper, actionable intelligence, not just antivirus verdicts.

---

## Our Solution

A two-stage analysis pipeline that combines static triage with full dynamic execution in a sandboxed Android environment, culminating in a C2 infrastructure map and investigator-friendly threat report.

```
Upload APK → Static Triage → Decision Engine → Dynamic Execution → C2 Map + Report
```

---

## Pipeline Overview

### Stage 1 — Static Triage
Runs inside a hardened Docker container (gVisor + Seccomp + AppArmor) on GCP. The APK is not executed — only read and analyzed.

| Tool | Role |
|------|------|
| **APKiD** | Fingerprints the APK — detects anti-debug, anti-emulator, anti-tamper protections and obfuscation |
| **jadx** | Decompiles to Java — scans for hardcoded IPs/domains, suspicious permissions, malicious patterns |
| **APKTool** | Decompiles to Smali — lower-level inspection of evasion logic |
| **Decision Engine** | Scores the APK — clean samples get a lightweight report, suspicious ones proceed to Stage 2 |

### Stage 2 — Dynamic Deep Scan
Runs directly on the GCP instance via Waydroid (full Android environment). Anti-analysis protections are patched before execution. The APK runs freely so its true behavior is revealed.

| Tool | Role |
|------|------|
| **APKTool + uber-apk-signer** | Patches anti-analysis checks, resigns the APK |
| **Waydroid** | Full Android runtime — APK believes it is on a real device |
| **Frida + Objection** | Runtime instrumentation — intercepts API calls, logs behavior, spoofs device identity |
| **mitmproxy** | Network capture and C2 sinkhole — fake C2 server responds, full traffic logged |
| **Clock acceleration** | Jumps system time forward to trigger time-bomb malware |

## What the Platform Outputs

### C2 Infrastructure Map
The core differentiator. A visual network graph (D3.js / vis.js) linking:
- Malicious APK nodes
- C2 server IPs and domains
- Infrastructure overlaps across analyzed samples
- Connected URLs and endpoints

Gives investigators a picture of the broader criminal operation — not just a single file verdict.

### IOC Table
Structured indicators extracted from both stages:
- Contacted IPs, domains, and URLs
- Abused permissions
- Suspicious API calls and hardcoded identifiers
- Exfiltration indicators

### LLM Threat Report
Plain-English summary generated by an LLM — written for investigators, not engineers. Includes what the APK does, what it tries to steal, which servers it contacted, risk verdict, and recommended action.

### Exportable Case File
Human-readable report + IOC CSV/JSON + captured traffic summary — ready for case handling.

## Architecture

```
GCP Instance (outer wall)
├── Docker + gVisor + Seccomp + AppArmor
│   └── Stage 1: APKiD + jadx + APKTool + Decision Engine
└── Waydroid (directly on GCP)
    └── Stage 2: Frida + mitmproxy + Clock Acceleration
```
